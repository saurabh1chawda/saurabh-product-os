# Resume Philosophy

## Executive Summary

Resume OS treats the resume as a routing document, not a complete career archive.

The resume should help a recruiter, hiring manager, ATS, or referrer decide whether the candidate is relevant for a specific role. It should select evidence from Product OS, not duplicate Product OS.

## Operating Principles

| Principle | Meaning |
| --- | --- |
| Every resume answers why this company should interview this candidate | The resume must make role fit obvious, not merely list experience |
| Relevance is selected, not invented | Tailoring means choosing the right evidence, not creating new claims |
| Metrics must remain traceable | Every metric must map to canonical evidence or approved wording |
| Product OS supports the resume without overwhelming it | Use Product OS as proof, not as a substitute for resume relevance |
| Language should reflect the role without copying the JD | Mirror the role's problem space, not its exact phrasing |
| Every bullet should communicate scope, decision, action, or outcome | Avoid activity-only bullets |
| ATS optimization must not reduce human readability | Keywords help only when the resume still reads naturally |
| Company-specific projects must be clearly labeled | Simulations, case studies, prototypes, and open-source work must not be presented as employer production work |
| Leadership claims must match actual authority and scope | Do not imply people management, budget ownership, or executive authority without evidence |
| Technical depth must remain explainable | Any technical claim should be defendable in an interview |
| Two pages maximum by default | Extend only when a role explicitly requires more detail |
| The resume must be skimmable in 10 to 15 seconds | Headline, summary, titles, companies, metrics, and skills should be visible quickly |
| Evidence is stronger than adjectives | Replace claims like "strategic" with decisions, trade-offs, outcomes, and metrics |
| No unsupported metrics, technologies, responsibilities, or outcomes | If it cannot be defended, it does not belong |

## Anti-Principles

| Anti-Principle | Why It Is Dangerous |
| --- | --- |
| Keyword stuffing | Reduces credibility and can hurt recruiter readability |
| Generic summaries | Waste the most valuable attention window |
| Repeating the JD | Signals weak original positioning |
| Inflated scope | Creates interview and reference risk |
| Fabricated AI experience | Damages trust and is easily exposed in technical interviews |
| Duplicating Product OS content | Turns the resume into a portfolio summary instead of a role-fit document |
| Excessive links | Creates decision fatigue and can impair ATS parsing |
| Visual elements that impair ATS parsing | Reduce machine readability and risk dropped content |
| Claiming prototypes as production deployments | Misrepresents proof-of-work |
| Adding confidential employer information | Creates legal, ethical, and reputational risk |

## Evidence Hierarchy

| Evidence Type | Strength | Resume Use |
| --- | --- | --- |
| Verified business metric | Highest | Use directly when approved |
| Public Product OS brief | High | Use as supporting proof or interview hook |
| Open GitHub artifact | High | Use for frameworks, templates, and technical credibility |
| Role responsibility | Medium | Use when it clarifies scope |
| Project simulation | Medium | Use only when labeled clearly |
| Tool familiarity | Low | Use in skills, not as a headline claim |
| Unverified adjective | Lowest | Avoid |

## Resume Voice

Resume language should be:

- Specific
- Measured
- Senior
- Evidence-backed
- Human-readable
- ATS-safe
- Interview-defensible

Resume language should not be:

- Promotional
- Overloaded with buzzwords
- Overconfident without proof
- Written like a Product OS page
- Written like a job description copy-paste

## Company-Specific Adaptation Rules

Company-specific adaptation is valid when it changes:

- Headline framing
- Summary emphasis
- Selected bullets
- Skills order
- Project framing
- Domain terminology
- Product OS proof link selection

It is not valid when it changes:

- Employment dates
- Titles
- Company names
- Verified metrics
- Scope of authority
- Production status
- Confidential details

## Reference Resume Pattern

The eBay resume demonstrates the intended pattern:

- Stable career history remains stable.
- Company-specific relevance is introduced through the headline, summary, skills, and project.
- Experience bullets are selected for monetization, loyalty, growth, marketplace, and product-led outcomes.
- The company-specific project is framed as proof-of-work, not employment history.

Resume OS should preserve that distinction across every tailored resume.
