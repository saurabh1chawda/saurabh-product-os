# Product OS Integration Rules

## Executive Summary

Product OS should strengthen a resume by giving recruiters and hiring managers a path to deeper proof.

It should not overwhelm the resume, displace relevant experience, or turn the resume into a product portfolio.

Baseline:

Header:

```text
Email | Phone | LinkedIn | GitHub | Product OS
```

Summary:

At most one concise Product OS sentence when it strengthens differentiation.

Body:

Use either a relevant company-specific project or a flagship Product OS block, not both by default unless the role strongly justifies it.

Experience:

Do not place repetitive evidence links beneath every role.

## Link Placement Rules

| Location | Rule |
| --- | --- |
| Header | Include Product OS and GitHub by default |
| Summary | One Product OS sentence maximum |
| Experience | Avoid links under every role |
| Projects | Include one live or GitHub link only when relevant |
| Skills | Do not link |
| Education | Do not link unless required |

## Product OS Link In Header

Use when:

- The resume is for product leadership, AI PM, platform PM, or senior PM roles.
- Public proof improves confidence.
- The resume has room for a clean evidence link.

Avoid when:

- The application portal strips hyperlinks.
- The role requires a very traditional resume.
- Too many links would reduce readability.

## GitHub Link In Header

Use when:

- The role values reusable frameworks, AI artifacts, technical documentation, or open product work.
- The target audience includes engineering leaders or AI product teams.

Avoid when:

- GitHub artifacts are not relevant.
- The resume already includes a more relevant Product OS page.

## Optional Product OS Summary Sentence

Use a concise sentence when Product OS strengthens differentiation.

Example pattern:

```text
Creator of Product OS, a public product leadership platform documenting AI, platform, growth, and decision-system work.
```

Rules:

- One sentence maximum.
- Do not let it replace role fit.
- Do not repeat it elsewhere.

## Dedicated Flagship Product Section

Include when:

- The role strongly values proof-of-work.
- A Product OS or GitHub artifact directly maps to the JD.
- The resume needs evidence beyond employment bullets.

Do not include when:

- It displaces stronger work experience.
- A company-specific project is more relevant.
- The role is conservative and prefers standard resume structure.

## Product Leadership Brief Links

Use when:

- The brief directly supports the role archetype.
- A hiring manager is likely to value decision depth.
- The link can be labeled clearly.

Examples:

- Platform role: Logix Product Leadership Brief.
- Payments role: Mahindra Comviva Product Leadership Brief.
- Growth role: Simplilearn Product Leadership Brief.
- Enterprise SaaS role: JoVE Product Leadership Brief.

## Company-Specific Project vs Flagship Product

| Use Company-Specific Project When | Use Flagship Product When |
| --- | --- |
| The company context is highly specific | The role values reusable thinking |
| Proof-of-work improves differentiation | Public artifact already maps well |
| The project can be clearly labeled | The artifact is more mature |
| The target company problem is obvious | The resume needs credibility, not customization |

Default: choose one.

## Maximum Number Of Links

Recommended maximum:

- Header: 3 evidence/contact links beyond email and phone
- Body: 1 to 2 additional links
- Total: 5 or fewer links

Too many links create decision fatigue.

## Link Labeling Rules

Use readable labels:

- Product OS
- GitHub
- Executive Brief
- AI Product Playbook
- Logix Product Leadership Brief

Avoid:

- Raw long URLs in dense sections
- Ambiguous labels like "click here"
- Repeated links to the same destination

## ATS-Safe Hyperlink Practices

- Keep visible link labels meaningful.
- Ensure exported PDFs preserve clickable links.
- Include full URLs only when needed for plain-text parsing.
- Avoid embedded icons as the only link indicator.
- Verify every link in DOCX and PDF.

## Preventing Product OS From Displacing Experience

Product OS is supporting evidence. It should not replace:

- Employment history
- Business outcomes
- Leadership scope
- Role-specific skills
- Directly relevant bullets

If a Product OS block forces strong experience bullets off the resume, remove the Product OS block.

## Canonical URL Rules

Use canonical Product OS URLs only:

- `https://saurabh-product-os.vercel.app`
- `https://saurabh-product-os.vercel.app/executive`
- `https://saurabh-product-os.vercel.app/ai-product-playbook`
- `https://saurabh-product-os.vercel.app/case-studies`
- Relevant decision-system URLs when needed

Do not use temporary local paths, draft deployment URLs, or stale GitHub monorepo paths.
